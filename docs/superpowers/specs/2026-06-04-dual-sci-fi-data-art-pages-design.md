## 1. 目标

在 `筹码分布可视化` 目录中创建两件全新的前端艺术作品，用来展示网页创造力，而不是延续任何筹码分布、股票驾驶舱或数据看板题材。

核心要求:

- 风格是科幻数据艺术
- 一个 3D 页面，一个 2D 页面
- 两个页面彼此独立，分别成立
- 可交互，且交互强度接近小游戏式探索
- 复杂度尽量高
- 页面内几乎没有文字
- 第一屏直接进入作品本体，不做落地页或说明页

## 2. 非目标

以下内容不在本次设计范围内:

- 不复用现有筹码、K 线、金融、股票语义
- 不把旧页面改皮肤后继续使用
- 不依赖重文本说明帮助用户理解
- 不先做单页再把另一页降级成附属 demo

## 3. 总体方案

这次交付是一组并列作品，采用同一文明语系，但不共享画面结构，不互相依赖才能成立。

两个入口页:

- `void-temple.html`: 3D 探索式空间装置
- `signal-altar.html`: 2D 操纵式活体控制台

整体体验关系:

- 3D 页负责“走进去，唤醒空间”
- 2D 页负责“站在界面前，改写机器”
- 两页都包含明显的阶段推进，而非单次点击特效

## 4. 3D 页面设计: `void-temple.html`

### 4.1 体验定位

一个可以探索的异星圣殿/机器空间。用户进入后处在深色空腔内，中央是一枚会变形的核心，四周分布若干可触发装置。页面重点不是读信息，而是探索空间状态如何被自己改写。

### 4.2 空间构成

主要由以下视觉系统组成:

- 中央核心体: 呼吸、旋转、分层展开
- 环轨系统: 围绕核心的悬浮环与轨道组件
- 碑体节点: 可激活的关键结构
- 能量桥与门: 随阶段开启的新路径
- 粒子瀑布与星尘层: 用于强化深度与空间流动
- 体积雾与镜面地表: 提升仪式感和反射层次
- 外部深空层: 最终阶段揭示的远景宇宙结构

### 4.3 交互闭环

初始状态为低功率沉睡态。用户通过移动、观察、靠近和触发推进世界状态:

1. 进入场景，核心仅有低频呼吸
2. 接近装置后，路径和局部结构开始响应
3. 激活多个关键碑体后，几何结构发生重排
4. 新的区域、桥或门被打开
5. 达到最终阶段后，整座圣殿像反相展开，暴露外部深空层

交互结果必须改变场景组织本身，而不只是叠加发光或粒子。

### 4.4 输入方式

- 桌面端: 鼠标控制视角，键盘控制移动
- 移动端: 拖拽视角，缓速漂移式前进或聚焦式导航
- 靠近目标时可使用吸附式触发，减少精确瞄准带来的挫败感

## 5. 2D 页面设计: `signal-altar.html`

### 5.1 体验定位

一个“活体仪式控制台”。它看起来像未来装置的平面核心界面，不是普通 dashboard，而是一块会被操纵、增殖、过载、重组的发光机械图谱。

### 5.2 画面构成

主要由以下视觉系统组成:

- 中心相位盘: 可拖动并驱动全局相位变化
- 同心环与线路层: 构成主要骨架
- 节点阵列: 可点亮、连通、断开
- 波池与干涉纹: 随状态产生不同频谱响应
- 扫描束与游走信号: 提供仪器感和方向性
- 隐藏图层: 随探索逐步显现的复杂结构
- 最终机械星图: 界面完全觉醒后的全屏态

### 5.3 交互闭环

初始状态是相对克制的精密控制台。用户通过点、拖、连线、长按推动界面异化:

1. 拖动中心相位盘，带动整页旋转、配色偏移和波场变频
2. 点亮外围节点，激活回路、扫描束、干涉纹
3. 在节点之间拖出连线，生成临时能量网络
4. 在特定区域长按进入过载态，释放时触发整体重组
5. 隐藏层被掀开，界面扩张为活着的机械星图

最终效果应让用户感觉自己不是在点击按钮，而是在改写一台二维机器的规则。

## 6. 共通视觉语言

两页属于同一套审美系统，但分工明显:

- 3D 页偏体积、深度、空间仪式感
- 2D 页偏精密、密集、神经系统感

统一原则:

- 基底色接近黑色，保证沉浸和辉光对比
- 不使用单一蓝紫色系
- 主能量色采用多相位组合，例如电青、洋红、琥珀金、酸绿
- 颜色随状态机推进发生迁移
- 几乎不出现解释文字，只保留极少符号化控件

## 8. 技术路线

### 8.1 3D 页面

- 使用 `Three.js` 构建真实 3D 场景
- 通过程序生成几何、粒子、轨道、光环和装置结构
- 使用状态机驱动阶段变化，而不是散乱时间轴特效

### 8.2 2D 页面

- 主画面使用 `Canvas` 渲染高密度动态内容
- 必要的少量交互控件可叠加 DOM 或 SVG
- 线路、波纹、节点、扫描和辉光效果优先用代码生成

### 8.3 共同原则

- 体验尽量依赖程序化视觉，而非素材拼贴
- 页面默认全屏沉浸式布局
- 控件只保留声音、重置、全屏等少量符号按钮

## 9. 状态机设计

两页都采用离散阶段推进，避免用户操作缺乏反馈。

建议每页至少包含四个层级:

1. Dormant: 沉睡或待机态
2. Awakening: 局部响应，结构开始启动
3. Reconfiguration: 世界结构被改写
4. Ascended: 完整觉醒，进入高复杂度终态

阶段推进条件:

- 3D 页由探索与装置激活数触发
- 2D 页由节点激活、连线数、过载行为触发

## 10. 性能与适配边界

优先保证桌面端表现，同时为移动端做自适应降载。

桌面端目标:

- 首屏沉浸
- 交互流畅
- 阶段转换清晰

移动端策略:

- 自动降低粒子密度
- 自动降低后处理和辉光层数
- 控制输入方式，避免复杂键鼠逻辑硬搬到触屏

实现侧重点:

- 3D 页控制 draw call、实例化重复体、限制昂贵后处理
- 2D 页采用分层绘制，降低每帧重画成本

## 11. 验证标准

完成后用本地静态服务进行双页面验证，至少覆盖桌面与移动视口。

重点验证项:

- 首屏是否足够强烈，打开即进入作品
- 用户是否不读说明也能自然开始探索
- 阶段变化是否清楚，且能感知到“世界被推进”
- 3D 与 2D 两页是否各自成立，而不是一强一弱
- 页面是否保持几乎无文字的设计约束
- 性能是否稳定，无明显掉帧或交互卡顿

## 12. 实施边界

本次实现将以“高完成度网页艺术装置”为目标，而不是只做概念草图。允许多文件结构，允许引入前端运行所需的公共库，但应保持整体为本地可直接访问的静态作品集形态。

如果后续实现中需要在“极致视觉复杂度”和“交互稳定性”之间做取舍，优先保证:

1. 体验完整
2. 场景状态变化明确
3. 视觉密度足够高
4. 性能在常见设备上可接受

## 13. 交付结果定义

- 一个完整的 3D 页面 `void-temple.html`
